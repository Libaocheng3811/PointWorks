# PointWorks 自动测试模块设计方案

## 1. 概述

### 1.1 目标

为 PointWorks 项目引入系统化的自动测试体系，覆盖核心数据结构、算法模块、文件 I/O 三大libs层，实现：

- **算法正确性验证**：确保滤波、配准、分割、法线估计等算法在各种输入下行为正确
- **回归保护**：代码变更后自动检测是否引入已有功能的回归问题
- **性能基准**：对关键算法建立性能基线，检测性能退化
- **开发提效**：替代手动测试流程，缩短迭代周期

### 1.2 测试范围

| 层级 | 模块 | 可测试性 | 优先级 |
|------|------|----------|--------|
| libs/core | Cloud, CloudBlock, Octree, CloudType | 高（纯数据结构） | P0 |
| libs/algorithm | Filters, Registration, Segmentation, Normals, Features, Keypoints, Surface, DistanceCalculator | 高（静态方法，无UI依赖） | P0 |
| libs/io | FileIO, ProjectFile | 中（依赖文件系统和信号槽） | P1 |
| libs/viz | CloudView, OctreeRenderer | 低（依赖VTK渲染） | P2 |
| src/ | 工具/插件/编辑 | 低（依赖Qt UI） | P3 |

**核心原则**：优先测试 libs/ 层的纯逻辑代码，该层不依赖 Qt 事件循环和 VTK 渲染，测试成本最低、收益最高。

## 2. 技术选型

### 2.1 测试框架：Google Test (GTest) + Google Mock (GMock)

**选型理由**：

| 维度 | GTest | Catch2 | Boost.Test |
|------|-------|--------|------------|
| 成熟度 | 工业界标准，文档丰富 | 轻量但生态较小 | 重量级，需Boost |
| CMake 集成 | `FetchContent` 一行引入 | 同样支持 | 需完整Boost |
| 断言丰富度 | 极丰富（ASSERT/EXPECT） | 中等 | 丰富 |
| Mock 支持 | GMock 原生支持 | 无（需第三方） | 有Mock |
| 参数化测试 | 原生支持 | 原生支持 | 原生支持 |
| 与 PCL 兼容性 | PCL 自身测试用 GTest | - | - |

**结论**：GTest 与 PCL 生态一致，C++17 兼容，`FetchContent` 集成零配置。

### 2.2 测试数据方案

| 类型 | 说明 | 存放位置 |
|------|------|----------|
| 合成数据 | 代码内生成（平面、球体、立方体点云） | 测试代码内 |
| 固定 fixture | 小型二进制 PCD/LAS 文件（< 1MB） | `tests/data/` |
| 大数据性能测试 | 可选的大型点云文件 | `tests/data/benchmark/`（不加入 Git） |

**原则**：优先使用合成数据（自包含、确定性），少量 fixture 用于文件 I/O 测试。

## 4. CMake 集成

### 4.1 主 CMakeLists.txt 修改

在根 `CMakeLists.txt` 末尾新增：

```cmake
# ============ 测试支持 ============
option(BUILD_TESTS "Build unit tests" ON)

if(BUILD_TESTS)
    enable_testing()
    add_subdirectory(tests)
endif()
```

### 4.2 tests/CMakeLists.txt（总入口）

```cmake
include(FetchContent)

# 下载 Google Test
FetchContent_Declare(
    googletest
    GIT_REPOSITORY https://github.com/google/googletest.git
    GIT_TAG        v1.14.0
)
set(gtest_force_shared_crt ON CACHE BOOL "" FORCE)  # MSVC /MD 兼容
FetchContent_MakeAvailable(googletest)

# 测试公共工具库
add_subdirectory(common)

# 注册各模块测试
add_subdirectory(core)
add_subdirectory(algorithm)
add_subdirectory(io)

# 通过 CTest 注册所有测试
include(GoogleTest)
```

### 4.3 各模块 CMakeLists.txt 模板

以 `tests/algorithm/CMakeLists.txt` 为例：

```cmake
file(GLOB TEST_SRCS CONFIGURE_DEPENDS "${CMAKE_CURRENT_SOURCE_DIR}/*.cpp")

add_executable(test_algorithm ${TEST_SRCS})

target_link_libraries(test_algorithm
    PRIVATE
        ct_algorithm        # 被测模块
        ct_core
        test_common         # 测试公共工具
        GTest::gtest
        GTest::gtest_main
        ${PCL_LIBRARIES}
        ${VTK_LIBRARIES}
)

target_include_directories(test_algorithm PRIVATE
    ${CMAKE_SOURCE_DIR}/libs
)

# 注册到 CTest
gtest_discover_tests(test_algorithm)
```

### 4.4 编译与运行命令

```bash
# 构建测试（默认开启）
cmake -B build -S . -DCMAKE_BUILD_TYPE=Debug
cmake --build build --config Debug --target test_algorithm

# 运行所有测试
cd build && ctest --output-on-failure

# 运行单个模块测试
./build/bin/test_algorithm --gtest_filter="FiltersTest.*"

# 运行单个测试用例
./build/bin/test_algorithm --gtest_filter="FiltersTest.VoxelGrid*

# 输出 XML 报告（供 CI 使用）
./build/bin/test_algorithm --gtest_output=xml:test_results.xml

# 仅构建不构建测试
cmake -B build -S . -DBUILD_TESTS=OFF
```

### 5.2 自定义断言宏 (macros.h)

```cpp
#pragma once
#include <gtest/gtest.h>

// 断言 Cloud 非空
#define ASSERT_CLOUD_NOT_NULL(cloud) \
    ASSERT_NE(nullptr, cloud) << "Cloud pointer is null"

// 断言 Cloud 包含至少 N 个点
#define ASSERT_CLOUD_SIZE_AT_LEAST(cloud, n) \
    ASSERT_CLOUD_NOT_NULL(cloud); \
    ASSERT_GE(cloud->size(), (size_t)(n)) << "Expected >= " << (n) << " points, got " << cloud->size()

// 断言 Cloud 包含恰好 N 个点
#define ASSERT_CLOUD_SIZE_EQ(cloud, n) \
    ASSERT_CLOUD_NOT_NULL(cloud); \
    ASSERT_EQ(cloud->size(), (size_t)(n)) << "Expected " << (n) << " points, got " << cloud->size()

// 断言操作成功
#define ASSERT_SUCCESS(result) \
    ASSERT_TRUE(result.success) << "Operation failed: " << result.error_msg
```

## 7. 测试代码示例

### 7.1 滤波测试示例 (test_filters.cpp)

```cpp
#include <gtest/gtest.h>
#include <filters.h>
#include "test_helpers.h"
#include "macros.h"

using namespace ct;

class FiltersTest : public ::testing::Test {
protected:
    void SetUp() override {
        // 生成测试数据：1000 个 XY 平面上的点
        cloud_ = test_helpers::makePlane(1000, 10.0f, 10.0f);
        ASSERT_CLOUD_NOT_NULL(cloud_);
    }

    Cloud::Ptr cloud_;
};

// ========== PassThrough ==========

TEST_F(FiltersTest, PassThrough_XAxis) {
    auto result = Filters::PassThrough(
        cloud_, "x", 2.0f, 8.0f, false, nullptr, nullptr);

    ASSERT_CLOUD_NOT_NULL(result.result_cloud);
    EXPECT_GT(result.result_cloud->size(), 0u);
    EXPECT_LT(result.result_cloud->size(), cloud_->size());

    // 验证所有剩余点的 x 坐标在 [2, 8] 范围内
    auto pcl_cloud = result.result_cloud->toPCL_XYZ();
    for (const auto& pt : pcl_cloud->points) {
        EXPECT_GE(pt.x, 2.0f);
        EXPECT_LE(pt.x, 8.0f);
    }
}

TEST_F(FiltersTest, PassThrough_Negative) {
    auto result = Filters::PassThrough(
        cloud_, "x", 2.0f, 8.0f, true, nullptr, nullptr);

    ASSERT_CLOUD_NOT_NULL(result.result_cloud);
    // negative 模式：保留范围外的点
    auto pcl_cloud = result.result_cloud->toPCL_XYZ();
    for (const auto& pt : pcl_cloud->points) {
        EXPECT_TRUE(pt.x < 2.0f || pt.x > 8.0f);
    }
}

// ========== VoxelGrid ==========
```

## 9. 最佳实践

### 9.1 测试命名规范

```
ClassName_TestMethodName_Scenario_ExpectedResult

例：
FiltersTest_PassThrough_XAxis_FiltersCorrectly
RegistrationTest_ICP_KnownTransform_ConvergesWithinThreshold
```

### 9.2 测试编写原则

1. **确定性**：每个测试独立运行，不依赖其他测试的执行顺序或副作用
2. **可重复**：相同输入总是产生相同结果（使用固定 seed）
3. **快速**：单个测试应在秒级完成；性能基准测试单独分类
4. **自包含**：不依赖外部文件（除非是专门的 I/O 测试）
5. **有意义的断言**：不只检查"不崩溃"，要验证输出正确性

### 9.3 参数化测试

对于需要测试多种参数组合的场景，使用 GTest 的参数化测试：

```cpp
// 滤波器参数化测试
struct FilterParam {
    float voxel_size;
    float expected_ratio;  // 期望保留的点数比例
};

class VoxelGridParamTest : public ::testing::TestWithParam<FilterParam> {};

TEST_P(VoxelGridParamTest, ReducesCorrectly) {
    auto cloud = test_helpers::makePlane(10000, 10.0f, 10.0f);
    auto p = GetParam();

    auto result = Filters::VoxelGrid(
        cloud, p.voxel_size, p.voxel_size, p.voxel_size,
        false, nullptr, nullptr);

    float ratio = (float)result.result_cloud->size() / cloud->size();
    EXPECT_NEAR(ratio, p.expected_ratio, 0.05f);
}

INSTANTIATE_TEST_SUITE_P(VariousSizes, VoxelGridParamTest,
    ::testing::Values(
        FilterParam{0.1f, 0.90f},
        FilterParam{0.5f, 0.15f},
        FilterParam{1.0f, 0.04f}
    ));
```

## 10. 风险与注意事项

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| FileIO 依赖 Qt 信号槽 | 测试需要 QCoreApplication | 测试 main 中创建 `QCoreApplication` |
| PCL 内存开销大 | 大数据测试可能 OOM | 使用合成小数据，大数据测试独立进程 |
| CSF/3rdparty 算法非确定性 | 结果可能因平台/浮点差异不同 | 使用宽松容差（5%），不比较精确值 |
| GTest 与 MSVC /MD 冲突 | 链接错误 | 设置 `gtest_force_shared_crt ON` |
| 测试数据 Git 体积 | 小 PCD 文件可接受 | 大文件使用 Git LFS 或不加入版本控制 |

### FileIO 测试的 Qt 事件循环处理
