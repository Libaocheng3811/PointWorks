## 1. 设计概览

PointWorks 采用 **AOS (Array of Structures)** 的数据组织方式，通过 **八叉树空间索引** 将海量点云分割为多个 `CloudBlock`，每个 Block 存储实际点数据。整体架构分为三层：

```
┌─────────────────────────────────────────────────────┐
│                    Cloud (顶层)                       │
│  ┌─────────────────────────────────────────────────┐ │
│  │              OctreeNode (八叉树)                  │ │
│  │  ┌──────────┐  ┌──────────┐     ┌──────────┐   │ │
│  │  │Internal  │  │Internal  │ ... │Internal  │   │ │
│  │  │Node      │  │Node      │     │Node      │   │ │
│  │  │(LOD点集) │  │(LOD点集) │     │(LOD点集) │   │ │
│  │  └──┬──┬───┘  └──────────┘     └──────────┘   │ │
│  │     │  │                                       │ │
│  │  ┌──▼──▼───────────────────────────────────┐   │ │
│  │  │         Leaf Node                        │   │ │
│  │  │  ┌────────────────────────────────────┐  │   │ │
│  │  │  │      CloudBlock (数据块)            │  │   │ │
│  │  │  │  · m_points (XYZ坐标)              │  │   │ │
│  │  │  │  · m_colors (RGB颜色, 可选)        │  │   │ │
│  │  │  │  · m_normals (压缩法线, 可选)      │  │   │ │
│  │  │  │  · m_scalar_fields (标量场)         │  │   │ │
│  │  │  │  · m_vtk_polydata (VTK渲染缓存)    │  │   │ │
│  │  │  └────────────────────────────────────┘  │   │ │
│  │  └──────────────────────────────────────────┘   │ │
│  └─────────────────────────────────────────────────┘ │
│                                                       │
│  m_scalar_cache   (Cloud级标量场缓存)                 │
│  m_cached_xyz     (PCL XYZ缓存)                      │
│  m_cached_xyzrgb  (PCL XYZRGB缓存)                   │
│  m_cached_xyzrgbn (PCL XYZRGBN缓存)                  │
└─────────────────────────────────────────────────────┘
```

### 设计哲学

| 原则 | 实现方式 |
|------|---------|
| **AOS 存储** | 每个点作为一个整体存储，连续内存布局，缓存友好 |
| **延迟分配** | 颜色、法线使用 `unique_ptr`，无数据时不分配 |
| **自适应配置** | 根据点数自动决定是否启用八叉树、Block 大小、LOD 密度 |
| **类型擦除** | VTK 缓存通过 `shared_ptr<void>` 避免核心层编译依赖 VTK |
| **数据压缩** | 法线使用球面坐标编码 (2B)，而非存储 3 个 float (12B) |

### 2.3 CompressedNormal (压缩法线)

```cpp
struct CompressedNormal {
    uint16_t data = 0;

    void set(const Eigen::Vector3f& n);  // 3D向量 → 2B编码
    Eigen::Vector3f get() const;          // 2B编码 → 3D向量
};
```

| 字段 | 类型 | 大小 |
|------|------|------|
| data | uint16_t | 2 B |

#### 2.3.1 不压缩时的法线存储

PCL 的法线存储为 3 个 float + 1 个 curvature：

```
pcl::Normal
┌──────────┬──────────┬──────────┬───────────┐
│  float   │  float   │  float   │   float   │
│ normal_x │ normal_y │ normal_z │ curvature │
│   4 B    │   4 B    │   4 B    │    4 B    │
└──────────┴──────────┴──────────┴───────────┘
                    合计: 16 字节/法线
```

实际有效数据只有 3 个方向分量 (12B)，`curvature` 通常不用但占 4B。而且法线是**单位向量**（长度恒为 1），理论上只需要 2 个自由度即可表示。

#### 2.3.2 压缩原理：球面坐标编码

核心思想：**3D 单位向量 → 球面坐标 (θ, φ) → 量化为整数 → 打包到 16 bit**

```
单位向量 (x, y, z)           球面坐标 (θ, φ)           量化整数           打包
    │                          │                       │                │
    │  转换为球面坐标          │  线性量化              │  位拼接         │
    ▼                          ▼                       ▼                ▼
  x = sin(φ)·cos(θ)     θ ∈ [0, 2π]          theta → 9 bits       data = (θ<<7) | φ
  y = sin(φ)·sin(θ)     φ ∈ [0, π]            phi   → 7 bits       16 bit = 2 字节
  z = cos(φ)
```

16 bit 的位布局：

```
  bit 15  ...  bit 7  |  bit 6  ...  bit 0
  ┌───────────────────┬───────────────────┐
  │   theta (9 bits)  │   phi (7 bits)    │
  │   方位角 0~511    │   极角 0~127      │
  └───────────────────┴───────────────────┘
         高 9 位              低 7 位
```

为什么是 9+7 而不是 8+8：θ 范围 [0, 2π] 是 φ [0, π] 的两倍，分配更多 bit 量级合理。

#### 2.3.3 编码过程 (set)

```cpp
void set(const Eigen::Vector3f& n) {
    // 1. 归一化
    float len = n.norm();
    if (len < 1e-6f) { data = 0; return; }  // 零向量特殊标记
    Eigen::Vector3f normalized = n / len;

    // 2. 笛卡尔 → 球面坐标
    float phi   = acos(clamp(normalized.z(), -1, 1));     // 极角 [0, π]
    float theta = atan2(normalized.y(), normalized.x());   // 方位角 [-π, π]
    if (theta < 0) theta += 2 * M_PI;                      // 转为 [0, 2π]

    // 3. 量化：连续值 → 离散整数
    uint16_t theta_bits = (uint16_t)(theta / (2π) * 511);  // 9 bits, 0~511
    uint16_t phi_bits   = (uint16_t)(phi   / π   * 127);   // 7 bits, 0~127

    // 4. 位拼接
    data = (theta_bits << 7) | phi_bits;
}
```

#### 2.3.4 解码过程 (get)

```cpp
Eigen::Vector3f get() const {
    // 1. 位拆分
    uint16_t theta_bits = (data >> 7) & 0x1FF;   // 取高 9 位
    uint16_t phi_bits   =  data       & 0x7F;    // 取低 7 位

    // 2. 反量化：整数 → 连续值
    float theta = theta_bits / 511.0f * 2π;
    float phi   = phi_bits   / 127.0f * π;

    // 3. 球面 → 笛卡尔
    float sin_phi = sin(phi);
    return Vector3f(
        sin_phi * cos(theta),
        sin_phi * sin(theta),
        cos(phi)
    );
}
```

#### 2.3.5 精度分析

| 分量 | bit 数 | 级数 | 角度分辨率 |
|------|--------|------|-----------|
| θ (方位角) | 9 | 512 | 2π/512 ≈ **0.70°** |
| φ (极角) | 7 | 128 | π/128 ≈ **1.41°** |

最大角度偏差 ≈ 1-2°。渲染法线光照效果肉眼不可见差异；精确计算（如法线一致性检查、PCA）时需解码为 float3。

#### 2.3.6 压缩效果

```
               不压缩              压缩后            节省
             ┌─────────┐       ┌───────┐
每个法线      │ 16 字节  │       │ 2 字节│       14 字节
             │(3 float │       │(uint16│
             │+curv)   │       │  _t)  │
             └─────────┘       └───────┘
压缩率                           8:1
```

实际场景中的内存节省：

| 点数 | 不压缩 (16B/法线) | 压缩后 (2B/法线) | 节省 |
|------|-------------------|-------------------|------|
| 100万 | 16 MB | 2 MB | **14 MB** |
| 1000万 | 152 MB | 19 MB | **133 MB** |
| 1亿 | 1.49 GB | 186 MB | **1.30 GB** |
| 5亿 | 7.45 GB | 931 MB | **6.52 GB** |

对于带法线的点云，仅法线压缩一项就能让总内存降低约 **40-50%**（法线从不压缩 16B 变成 2B，从整个点的 XYZRGBN=21B 中省掉 14B）。

## 3. CloudBlock — 数据块

文件：`libs/core/octree.h`

`CloudBlock` 是八叉树叶子节点的数据负载，存储一个连续子空间内的所有点。

### 3.1 数据成员

```cpp
class CloudBlock {
public:
    using Ptr = std::shared_ptr<CloudBlock>;

    // === 核心数据 ===
    std::vector<pcl::PointXYZ> m_points;                              // 必需，XYZ坐标
    std::unique_ptr<std::vector<ColorRGB>> m_colors;                 // 可选，RGB颜色
    std::unique_ptr<std::vector<ColorRGB>> m_backup_colors;          // 可选，颜色备份
    std::unique_ptr<std::vector<CompressedNormal>> m_normals;        // 可选，压缩法线
    std::unordered_map<std::string, std::vector<float>> m_scalar_fields; // 标量场

    // 标量场指针缓存 (避免逐点哈希查找)
    std::vector<std::pair<std::string, std::vector<float>*>> m_scalar_ptrs;

    // === 空间属性 ===
    Box m_box;                                // 包围盒 (~104 B)

    // === 渲染状态 ===
    bool m_is_visible = true;
    bool m_is_dirty = true;                   // 脏标记
};
```

### 3.2 内存布局

每个 CloudBlock 的内存结构：

```
CloudBlock
├── m_points        : vector<PointXYZ>   →  header(24B) + N × 16B
├── m_colors        : unique_ptr<vector<ColorRGB>>  → 8B + header(24B) + N × 3B
├── m_backup_colors : unique_ptr<vector<ColorRGB>>  → 8B + header(24B) + N × 3B
├── m_normals       : unique_ptr<vector<CompressedNormal>> → 8B + header(24B) + N × 2B
├── m_scalar_fields : unordered_map<string, vector<float>>
│                     每个字段: key(~40B) + vector header(24B) + N × 4B
├── m_scalar_ptrs   : vector<pair<string, float*>>  → 每条目 ~40B
├── m_box           : Box (~104B)
├── m_is_visible    : bool (1B)
└── m_is_dirty      : bool (1B)
```

### 3.3 延迟分配策略

颜色和法线采用 `unique_ptr` 延迟分配，避免不必要的内存开销：

| 场景 | m_colors | m_normals | 内存节省 |
|------|----------|-----------|---------|
| XYZ 点云 | nullptr | nullptr | 100% 颜色/法线内存 |
| XYZRGB 点云 | 已分配 | nullptr | 100% 法线内存 |
| XYZRGBN 点云 | 已分配 | 已分配 | 0% |
| 启用标量场着色 | 已分配 + backup | nullptr | 法线内存 (backup_colors 按需分配) |

`m_backup_colors` 仅在调用 `backupColors()` 时分配，用于标量场着色时保存原始颜色以便恢复。

### 3.4 标量场指针缓存

```cpp
std::vector<std::pair<std::string, std::vector<float>*>> m_scalar_ptrs;
```

`addPoint()` 逐点插入时需要同步更新所有标量场。如果每次都查 `unordered_map`，哈希查找开销在批量插入时会显著累加。`m_scalar_ptrs` 预缓存了指针，将查找代价从 O(H) 降到 O(K)（K=标量场数量，通常很小）。

```cpp

```

叶子节点通过 `m_block != nullptr` 判断。分裂时，`m_block` 被置空，节点变为内部节点。

### 4.3 子节点索引计算

```cpp
int getChildIndex(const pcl::PointXYZ& pt) const {
    float cx = m_box.translation.x();
    float cy = m_box.translation.y();
    float cz = m_box.translation.z();
    int index = 0;
    if (pt.x >= cx) index |= 1;   // bit 0: X 方向
    if (pt.y >= cy) index |= 2;   // bit 1: Y 方向
    if (pt.z >= cz) index |= 4;   // bit 2: Z 方向
    return index;                  // 0~7
}
```

每个子节点的包围盒大小为父节点的 1/2，中心偏移由 `childIdx` 的 bit 位决定：

```
childIdx  dx    dy    dz
  0       -1    -1    -1    (左下后)
  1       +1    -1    -1    (右下后)
  2       -1    +1    -1    (左下前)
  3       +1    +1    -1    (右下前)
  4       -1    -1    +1    (左上后)
  5       +1    -1    +1    (右上后)
  6       -1    +1    +1    (左上前)
  7       +1    +1    +1    (右上前)
```

### 4.4 内存布局

```
OctreeNode
├── m_parent       : 指针 (8B)
├── m_children[8]  : 8 × unique_ptr (64B)
├── m_box          : Box (~104B)
├── m_depth        : int (4B)
├── m_total_points_in_node : size_t (8B)
├── m_block        : shared_ptr<CloudBlock> (16B)
├── m_lod_points   : vector<LODPoint> → header(24B) + M × 16B
├── m_lod_dirty    : bool (1B + padding)
└── 固定开销合计     : ~230B (不含 m_lod_points 数据)
```

### 5.2 关键设计决策

| 决策 | 原因 |
|------|------|
| `enable_shared_from_this` | 允许 Cloud 安全地创建指向自身的 shared_ptr，供回调/异步任务使用 |
| 禁用拷贝和移动 | 点云数据量大，拷贝代价极高；移动语义与 shared_ptr 子对象冲突 |
| 提供 `swap()` | 作为原子操作的基础，用于"构建新数据 → swap 替换"模式 |
| `m_all_blocks` 扁平列表 | 避免每次遍历八叉树，提供 O(B) 的 Block 迭代能力 |
| `m_last_insert_node` | 缓存上一次插入的叶子节点，对空间局部性好的数据加速插入 |
| PCL 缓存标记 `mutable` | 允许在 const 方法中延迟构建缓存 |

## 6. 八叉树分块策略

### 6.1 自适应配置 (`calculateAdaptiveConfig`)

根据总点数自动决策分块策略，分三个层级：

```
总点数 N
  │
  ├─ N < 1000万 → 直通模式 (Passthrough)
  │     enableOctree = false
  │     maxPointsPerBlock = N + 1000
  │     maxLODPoints = 0
  │     maxDepth = 0
  │     (所有点在一个 Block 中，不建八叉树)
  │
  └─ N >= 1000万 → 八叉树模式
        ├─ 计算 idealBlockSize = N / TARGET_BLOCK_COUNT (1024)
        │     钳位到 [MIN_BLOCK_SIZE(3万), MAX_BLOCK_SIZE(100万)]
        │
        ├─ 计算 maxLODPoints = idealBlockSize × LOD_POINT_RATIO (0.5)
        │     钳位到 [MIN_LOD_SIZE(1万), MAX_LOD_SIZE(10万)]
        │
        └─ maxDepth = 8 (默认)
```

**设计目标**：叶子节点数保持在 ~1024 个，平衡以下因素：
- 太少 (<100)：视锥剔除粒度粗，绘制不必要的点
- 太多 (>10000)：DrawCall 开销增大，Actor 管理复杂
- Block 太大 (>100万)：单次 GPU 上传卡顿
- Block 太小 (<3万)：节点管理开销大于数据本身

### 6.2 点的插入流程

```
addPoint(pt, color, normal)
  │
  ├─ 懒初始化: 若无根节点，创建默认包围盒的八叉树
  │
  └─ insertPointToOctree(root, pt, color, normal)
        │
        ├─ 当前是叶子节点?
        │     ├─ 未满 或 达到最大深度 → 直接 addPoint 到 Block
        │     └─ 已满且可分 → splitNode(node) → 递归重新插入
        │
        └─ 当前是内部节点?
              ├─ 1. 统计 m_total_points_in_node++
              ├─ 2. 蓄水池采样 LOD (如果 maxLODPoints > 0)
              ├─ 3. 计算 childIdx
              ├─ 4. 按需创建子节点 (包含包围盒计算)
              └─ 5. 递归进入子节点
```

### 6.3 节点分裂过程 (`splitNode`)

当一个叶子节点的 Block 达到 `maxPointsPerBlock` 时触发分裂：

```
splitNode(node)
  │
  ├─ 1. 取走旧 Block 的所有权: oldBlock = node->m_block; node->m_block = nullptr
  │     (节点从叶子变为内部节点)
  │
  ├─ 2. 预分配 LOD 容量: node->m_lod_points.reserve(maxLODPoints)
  │
  ├─ 3. 遍历旧 Block 的所有点:
  │     ├─ A. LOD 提取 (蓄水池采样)
  │     │     从旧点集中均匀抽取 maxLODPoints 个点保留在父节点
  │     │
  │     ├─ B. 子节点分发
  │     │     根据 getChildIndex(pt) 路由到 8 个子空间
  │     │     按需创建子节点，预分配内存
  │     │
  │     └─ C. 数据移动
  │           坐标、颜色、备份颜色、法线、标量场 全部同步迁移
  │
  ├─ 4. 清理旧块: oldBlock->clear()
  │
  └─ 5. 从 m_all_blocks 中移除空块
```

**关键细节**：
- 分裂时继承父节点的属性状态（颜色、法线是否启用）
- 子节点的 Block 同步注册所有标量场定义
- 如果旧块有 `m_backup_colors`，子块也会初始化备份容器
- 使用 `reserve()` 预分配避免频繁扩容

### 6.4 批量插入优化 (`addPoints`)

```cpp
void Cloud::addPoints(pts, colors, normals, scalars) {
    for (size_t i = 0; i < n; ++i) {
        // 快速路径：利用 m_last_insert_node 缓存
        // 如果上次插入的叶子节点未满且包含当前点，直接插入
        if (m_last_insert_node && 未满 && 在AABB内) {
            targetBlock->addPoint(p, c, nm);
        }
        // 慢速路径：完整递归插入
        else {
            targetBlock = insertPointToOctree(root, pts[i], c, nm);
            // 更新 m_last_insert_node 为当前叶子
        }

        // 补充标量数据 (如果有)
        targetBlock->registerScalarField(name);
        targetBlock->m_scalar_fields[name][idx] = val;
    }
}
```

`m_last_insert_node` 缓存上一次成功插入的叶子节点。对于空间局部性好的点云数据（如 LAS 按行扫描存储），命中率极高，避免每点都从根节点遍历八叉树。

### 6.5 特殊情况：`makeAdaptive` 修正

加载完成后的自适应配置修正：

```cpp
void Cloud::makeAdaptive() {
    CloudConfig config = calculateAdaptiveConfig(count);

    // 关键修正：如果树结构已经是分裂状态
    // 即使点数 < 1000万，也必须强制启用八叉树
    if (isStructureSplit()) {
        config.enableOctree = true;
        config.maxDepth = 8;
        config.maxPointsPerBlock = 60000;
    }

    // 全局 shrink_to_fit — 回收 reserve 造成的 capacity 浪费
    for (auto& block : m_all_blocks) {
        block->m_points.shrink_to_fit();
        if (block->m_colors) block->m_colors->shrink_to_fit();
        if (block->m_normals) block->m_normals->shrink_to_fit();
        for (auto& kv : block->m_scalar_fields) kv.second.shrink_to_fit();
    }
}
```

> `shrink_to_fit()` 很重要：加载时 `reserve(maxPointsPerBlock)` 预分配的内存，在最终 Block 不足时需要释放。

## 7. LOD 设计与实现

### 7.1 设计目标

在缩放浏览时，远处区域不加载精细数据，而是用少量代表点 (LOD) 替代，保证：
1. **缩小时不出现空洞** — 根节点和浅层节点需要足够多的 LOD 点
2. **内存可控** — 每个节点的 LOD 点数有上限
3. **均匀采样** — 蓄水池采样保证每个点被选中的概率相等

### 7.2 LOD 数据存储位置

```
OctreeNode 类型    LOD 存储方式
───────────────   ──────────────────────────────
叶子节点           无 LOD (直接渲染 Block 原始数据)
内部节点           m_lod_points (vector<LODPoint>)
                  每个点 16B: XYZ(12B) + RGB(3B) + pad(1B)
```

### 7.3 加载时的实时 LOD 采样 (蓄水池采样)

在 `insertPointToOctree()` 的内部节点路径中，每经过一个点就进行蓄水池采样：

```cpp
// 阶段 A: 蓄水池未满，直接添加
if (node->m_lod_points.size() < maxLODPoints) {
    node->m_lod_points.push_back(lod_pt);
}
// 阶段 B: 蓄水池已满，以 k/n 概率随机替换
else {
    size_t capacity = maxLODPoints;
    size_t current_n = node->m_total_points_in_node;

    // 生成 [0, n) 的随机数，若 < capacity 则替换
    if (random(0, current_n-1) < capacity) {
        node->m_lod_points[random(0, capacity-1)] = lod_pt;
    }
}
```

**蓄水池采样原理**：

```
设蓄水池容量 k = maxLODPoints, 当前已处理 n 个点
- 前 k 个点直接填入蓄水池
- 第 i > k 个点，以 k/i 的概率决定是否保留
  - 如果保留，随机替换蓄水池中的一个位置

数学性质：每个点被保留在蓄水池中的最终概率 = k/N (均匀，N 为总点数)
```

为什么每个点的概率都相同？

```
每个点的最终概率 = 它进入池子的概率 × 它之后每一步不被替换的概率

越早的点，进入容易但被替换的次数多
越晚的点，进入难但被替换的次数少
两者恰好抵消，结果都等于 k/N
```

具体例子（k=1, N=3）：

```
池子容量 1，总共 3 个点

── 点 1 ── 池空，直接入池
   池: [P1]

── 点 2 ── 以 1/2 概率替换
   池: [P1] 概率 1/2，[P2] 概率 1/2

── 点 3 ── 以 1/3 概率替换当前池中的点
   决策树:
              P1在池(1/2)          P2在池(1/2)
              ╱       ╲            ╱       ╲
          替换(1/3)  不换(2/3)  替换(1/3)  不换(2/3)
            │          │          │          │
           [P3]       [P1]       [P3]       [P2]

   P1 最终概率 = 1/2 × 2/3 = 1/3
   P2 最终概率 = 1/2 × 2/3 = 1/3
   P3 最终概率 = 1/2 × 1/3 + 1/2 × 1/3 = 1/3

   全部 = k/N = 1/3 ✓
```

### 7.4 分裂时的 LOD 提取

`splitNode()` 中，旧 Block 的数据被分发到子节点之前，先提取 LOD：

}

### 7.5 全局 LOD 重建 (`generateLOD`)

加载完成后调用 `generateLOD()`，自底向上重建所有内部节点的 LOD：

```cpp
void Cloud::generateLODRecursive(OctreeNode* node) {
    // 叶子节点：清空 LOD
    if (node->isLeaf()) {
        node->clearLOD();
        return;
    }

    // 先递归处理所有子节点（自底向上）
    for (int i = 0; i < 8; ++i)
        generateLODRecursive(node->m_children[i].get());

    // === 深度感知 LOD 容量 ===
    size_t target_lod_size;
    if (node->m_depth == 0) {
        // 根节点：4倍容量，最少 100,000 点
        target_lod_size = max(maxLODPoints * 4, ROOT_MIN_LOD_SIZE);
    } else if (node->m_depth <= 2) {
        // 浅层节点：按深度因子放大
        float depthFactor = 1.0f + (3.0f - node->m_depth) * 0.5f;
        target_lod_size = maxLODPoints * depthFactor;
        target_lod_size = max(target_lod_size, ABSOLUTE_MIN_LOD_PER_NODE);
    } else {
        // 深层节点：标准容量
        target_lod_size = maxLODPoints;
    }
    target_lod_size = min(target_lod_size, MAX_LOD_SIZE * 2);

    // === 收集候选点 ===
    for (每个子节点) {
        if (子节点是叶子) {
            // 从 Block 中等步长采样
            step = block_size / target_lod_size;
            for (k = 0; k < block_size; k += step)
                candidates.push_back(point);
        }
        else {
            // 直接继承子节点的 LOD 点
            candidates.insert(end, child.m_lod_points);
        }
    }

    // === 最小点数保障 ===
    if (candidates.size() < ABSOLUTE_MIN_LOD_PER_NODE) {
        // 补充采样：用 step/2 偏移量采样，避免与前一轮重复
        ...
    }

    // === 最终降采样 ===
    if (candidates.size() > target_lod_size) {
        shuffle(candidates);           // 随机打乱
        candidates.resize(target_lod_size);  // 截断到目标大小
    }

    node->m_lod_points = move(candidates);
    node->m_lod_points.shrink_to_fit();
}
```

**深度感知 LOD 容量表**：

| 节点深度 | 深度因子 | 容量倍率 | 说明 |
|---------|---------|---------|------|
| 0 (根) | — | 4× | 最少 100,000 点，缩小时覆盖全局 |
| 1 | 2.5 | 2.5× | 大面积覆盖 |
| 2 | 2.0 | 2.0× | 中等覆盖 |
| 3+ | 1.0 | 1.0× | 标准容量 |

> 设计动机：缩放视图时，根节点和浅层节点是视觉上最先被看到的，如果 LOD 点太少，会出现明显空洞。深层节点覆盖面积小，标准容量已足够。

### 7.6 LOD 颜色同步

当用户切换颜色模式（标量场着色、单色等）时，LOD 点的颜色也需要更新：

```cpp
void Cloud::updateLODColorRecursive(OctreeNode* node, const ColorRGB& rgb) {
    if (!node || node->m_lod_points.empty()) return;
    for (auto& p : node->m_lod_points) {
        p.r = rgb.r; p.g = rgb.g; p.b = rgb.b;

        updateLODColorRecursive(node->m_children[i].get(), rgb);
}
```

## 8. 渲染缓存机制

文件：`libs/viz/octreerenderer.h`, `libs/viz/octreerenderer.cpp`

### 8.1 SSE 遍历策略

`OctreeRenderer` 使用 **Screen Space Error (SSE)** 策略决定渲染精度：

```
每帧更新流程:
  1. 获取相机参数 (位置、方向、视锥面)
  2. 初始化优先队列，根节点入队
  3. while (队列非空):
       a. 取出屏幕投影最大的节点
       b. 视锥体剔除 → 不可见则跳过
       c. 预算检查 → 是否还有渲染预算
       d. 分裂决策:
          should_split = (screenSize > threshold) && hasChildren && budgetAllows
          特殊: LOD点过少时也尝试分裂 (空洞回退)
       e. 如果 should_split → 子节点入队
          否则 → 加入可见列表
  4. 差异化更新 Actor
```

### 8.2 动态阈值与点预算

| 状态 | 屏幕阈值 | 点预算 | 说明 |
|------|---------|--------|------|
| 首次加载 | 300 px | 500万 | 避免一次性创建过多 Actor |
| 交互中 | 120 px | 1000万 | 适中阈值，预算充足 |
| 静止 | 80 px | 配置值 (默认1000万) | 高质量渲染 |

### 8.3 Actor 缓存管理

```cpp
// 核心缓存
std::unordered_map<OctreeNode*, vtkSmartPointer<vtkActor>> m_actor_cache;

// 上一帧可见节点集
std::unordered_set<OctreeNode*> m_current_visible_nodes;

// 缓存驱逐
std::unordered_map<OctreeNode*, int> m_hidden_frames;  // 连续不可见帧数
size_t m_max_cached_actors = 500;                       // 缓存上限
static constexpr int EVICTION_FRAMES = 5;              // 连续隐藏多少帧后驱逐
```

**Actor 缓存策略**：

| 操作 | 条件 | 行为 |
|------|------|------|
| 创建 | 节点首次可见 | 创建 vtkPolyData + Actor，加入缓存 |
| 复用 | 节点在缓存中且仍可见 | 仅设置 Visibility=1 |
| 隐藏 | 节点不在新一帧可见列表 | 设置 Visibility=0，hidden_frames++ |
| 驱逐 | 缓存总数 > max_cached_actors 且 hidden_frames >= 5 | 从 VTK Renderer 移除 Actor，释放 GPU 资源 |

**自适应缓存上限**：

```cpp
size_t totalBlocks = m_cloud->getBlocks().size();
m_max_cached_actors = clamp(totalBlocks / 2, 200, 1500);
```

### 8.4 VTK 数据构建 (`getOrCreateActor`)

每个可见节点构建一个独立的 VTK 渲染管线：

```
vtkPoints (float[3] × N)
    ↓

    ↓
vtkPolyData = Points + Colors + VertexCells
    ↓

    ↓

```

### 10.4 内存热点排名

| 排名 | 组件 | 占比 (典型) | 优化潜力 |
|------|------|-----------|---------|
| 1 | Block 原始数据 (坐标) | ~40-50% | 低 (PCL对齐无法改) |
| 2 | VTK 渲染缓存 | ~20-30% | 中 (仅视锥内) |
| 3 | PCL 兼容缓存 | ~0-50% | **高** (按需触发，可及时释放) |
| 4 | LOD 数据 | ~5-15% | 中 (可压缩) |
| 5 | 八叉树元数据 | <0.1% | 无需优化 |

### 10.5 内存放大因子

以 XYZRGB 点云为例，每点的"有效载荷"仅为 XYZ(12B) + RGB(3B) = 15B，但实际存储：

| 层级 | 每点大小 | 放大倍率 | 原因 |
|------|---------|---------|------|
| Block 原始 | 19B | 1.27× | PointXYZ padding (16B vs 12B) |
| VTK 缓存 | 15B | 1.0× | 使用原始 float[3]，无padding |
| PCL XYZRGB | 32B | 2.13× | PCL 内部大量 padding |
| PCL XYZRGBN | 48B | 3.2× | 更极端的 padding |

**综合放大** (Block + VTK)：典型情况下每有效字节占用约 **2.27B** 内存 (19B + 15B / 15B)。

## 11. 优化方向

### 11.1 高优先级

| 问题 | 影响 | 建议方案 |
|------|------|---------|
| PCL 缓存驻留不释放 | 亿级点云触发后增加 2-3 GB | 使用后立即 `invalidatePCLCache()`；或改为按需生成不缓存 |
| 标量场双重存储 | Block + Cloud 各存一份 | 明确职责：Block 级为权威数据，Cloud 级仅作加速查询缓存 |
| backup_colors 全量复制 | 标量场着色时颜色翻倍 | 考虑延迟分配或仅备份有变化的 Block |

### 11.2 中优先级

| 问题 | 影响 | 建议方案 |
|------|------|---------|
| LODPoint 16B/点 | LOD 数据量大 | RGB 可压缩为 RGB565 (2B)，每点减至 14B |
| 法线解压膨胀 | VTK 缓存中法线 2B→12B (6×) | GPU 端解码或使用 VTK 自定义 Attribute |
| PointXYZ padding | 每点多占 4B (25%) | 长期：使用自定义 XYZ struct (12B)；短期：无法改 |

### 11.3 长期方向

| 方向 | 说明 |
|------|------|
| SOA 存储 | 坐标/颜色/法线分列存储，利于 SIMD 和 GPU 直传 |
| GPU 直通 | Block 数据零拷贝上传到 GPU，省去 VTK 中间层 |
| 内存映射文件 | 超大点云 (>10亿) 使用 mmap 按需加载 |
| 流式渲染 | 仅保留视锥内 Block 的 VTK 缓存，其余释放 |
